# Conventions
Bourbon is opinionated about where things live. Every folder has one job. This is what makes projects predictable, this lets the CLI & the deploy step understand your project automatically.



## Project structure
What `bourbon init` scaffolds:

```
my-app/
├── api/                      # Backend · runs on the server (Edge Functions)
│   └── posts.js
│
├── public/                   # Frontend (runs in the browser)
│   ├── views/                # One folder per view/page
│   │   └── dashboard/
│   │       ├── index.js
│   │       └── components/   # Components used only by this view
│   │
│   ├── components/           # Global components, used across views
│   ├── styles/
│   │   ├── tokens.css        # Design tokens (CSS custom properties)
│   │   └── global.css        # Resets and base styles
│   │
│   ├── utils/                # Pure frontend helpers (no network, no Supabase)
│   └── index.html            # App entry point
│
├── supabase/
│   ├── schemas/              # Your database, split by concern
│   │   ├── tables.sql
│   │   ├── rls.sql
│   │   ├── rpc.sql
│   │   ├── triggers.sql
│   │   └── seed.sql          # Local dev test data, auto-applied
│   │
│   └── migrations/           # Auto-generated (never edit by hand)
│
├── .output/                  # Generated by Bourbon (gitignored)
├── dist/                     # Vite build output (gitignored)
├── bourbon.config.js         # Supabase URL and keys
├── vite.config.js            # Generated; points Vite at public/
└── .env.local                # Secrets (gitignored)
```



## The core rule
```
api/         → runs on the server
public/      → runs in the browser
supabase/    → your database
```

If you can read the root folder, you understand the whole app.



## Components
All components live in `public/components/`.

What you do inside that folder (subfolders, naming, organization) is your choice. Bourbon only enforces that components don't live anywhere else.

`bourbon add component button` creates `public/components/button.js`.
`bourbon check` warns if a component is defined outside `public/components/`.



## Schema files
Split by lifecycle, because each changes at a different rate:
- `tables.sql`: table definitions (changes rarely)
- `rls.sql`: row level security policies (changes often)
- `rpc.sql`: stored procedures (changes with business logic)
- `triggers.sql`: database triggers (set and mostly forgotten)
- `seed.sql`: fake data for local dev, applied automatically

Keeping them separate means you always know where to look and changes are easy to review.



## Frontend helpers vs backend logic
- Pure logic with no side effects (formatting, sorting) → `public/utils/`, runs in the browser
- Anything touching the database or external services → `api/`, runs on the server

There is no "services" folder. In the typical vanilla-JS + Supabase setup, "services" are browser files that call Supabase directly with the anon key, which is the insecure pattern Bourbon removes. That work moves server-side into `api/`.
